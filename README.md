# Запуск проекта
1. Клонируйте репозиторий: git clone https://github.com/DJBlin11/Reporting-AI-Agent-Test-Case.git

3. cd Reporting-AI-Agent-Test-Case в терминале

4. Установите зависимости:
pip install -r requirements.txt

5. Запустите агента:
python agent.py

# Краткое описание
**1. Роль агента**
   
  Агент выполняет роль аналитического слоя, который преобразует сырые данные SaaS-системы в интерпретируемый бизнес-отчет. Его задача заключается не только в     расчёте показателей, но и в формировании смысловых выводов, объяснении причин изменений метрик и генерации рекомендаций на основе данных.

**3. Устройство агента**

Система построена как последовательный pipeline:
   
- Data generation (generator.py) создаёт синтетические SaaS-данные

- Validation layer (validator.py) проверяет целостность и корректность сырых данных

- Metrics calculation (metrics.py) детерминированно рассчитывает ключевые бизнес-метрики: revenue, churn rate, ARPU

- Metrics validation (metrics_validator.py) проверяет корректность уже вычисленных метрик

- Visualization (visualizer.py) строит графики для анализа трендов

- Report agent (report_agent.py) формирует структурированный prompt, передаёт данные в LLM, получает итоговый бизнес-отчет

- LLM client (llm_client.py) управляет вызовом модели, содержит retry-логику и обработку ошибок API

**4. Использование обычных вычислений**

Все численные расчёты выполняются вне LLM для обеспечения точности. В частности, метрики вроде monthly revenue, churn rate и ARPU считаются напрямую в коде, а не моделью. Это позволяет избежать ошибок генерации и гарантирует корректность финансовой логики.

**5. Prompts**

LLM используется исключительно для генерации аналитического отчета. Prompt задаёт строгую структуру отчёта (Executive Summary, Revenue Trend, Churn Trend, ARPU Trend и т.д.), а также требования к глубине анализа: объяснение причин изменений, использование количественных значений и форматирование в markdown. Дополнительно в prompt передаются таблица метрик и результаты всех проверок данных.

**6. Guardrails и ограничения**

В системе реализованы базовые уровни контроля качества. На уровне данных выполняется проверка корректности входного датасета (наличие колонок, отсутствие отрицательных значений и переплат), а на уровне метрик — проверка математической корректности рассчитанных показателей (например, churn rate не превышает 1 и отсутствуют некорректные значения). Также в LLM-клиенте добавлены retry-механизмы и обработка ошибок API, чтобы повысить устойчивость системы.

**7. Причины выбора подхода**

Выбран гибридный подход, в котором детерминированные вычисления отделены от генерации текста, обеспечивает баланс между точностью и гибкостью. Код гарантирует корректность чисел и воспроизводимость результатов, а LLM отвечает за интерпретацию и бизнес-аналитику. Такая архитектура близка к реальным production-пайплайнам в аналитических и fintech-системах, где важно одновременно сохранять надёжность расчётов и получать качественные инсайты.
